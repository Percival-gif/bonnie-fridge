# 04-代码规范

## 1. Python 后端规范

### 1.1 命名规范
- 模块名：`snake_case.py`（如 `kimi_service.py`）
- 类名：`PascalCase`
- 函数/变量：`snake_case`
- 常量：`UPPER_SNAKE_CASE`
- 路由前缀：`/api/模块名`（如 `/api/ingredients`）

### 1.2 文件组织
```
backend/
├── main.py              # 应用入口，只注册路由和中间件
├── routers/             # 路由层（处理 HTTP 请求/响应）
│   ├── ingredients.py
│   └── recipes.py
├── services/            # 服务层（业务逻辑、外部 API 调用）
│   └── kimi_service.py
└── requirements.txt     # 依赖锁定版本
```

### 1.3 错误处理
- 所有路由必须 `try/except` 捕获异常
- 返回统一 JSON 格式：`{success, data/error}`
- 异常打印 traceback，便于调试

### 1.4 注释规范
- 模块顶部写 docstring 说明用途
- 复杂函数写 Args/Returns 注释
- 关键业务逻辑加行内注释

## 2. 前端规范

### 2.1 命名规范
- ID：`kebab-case`（如 `upload-placeholder`）
- 类名：`kebab-case`
- JS 变量/函数：`camelCase`
- 全局常量：`UPPER_SNAKE_CASE`

### 2.2 文件组织
```
frontend/
├── index.html           # 单页应用入口
├── css/
│   └── style.css        # 全局样式 + 组件样式
└── js/
    └── app.js           # 所有前端逻辑（MVP 阶段单文件）
```

### 2.3 API 调用规范
- 所有请求集中使用 `API_BASE` 前缀
- 异步函数统一 `try/catch`
- 错误提示统一调用 `showToast()`

### 2.4 状态管理
- MVP 阶段使用全局变量（`currentIngredients`, `shoppingList`）
- 避免直接操作 DOM，使用渲染函数（`renderIngredients()`）

## 3. Git 提交规范

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat:` | 新功能 | `feat: 添加语音补充食材功能` |
| `fix:` | 修复 bug | `fix: 修复图片超过 10MB 崩溃问题` |
| `docs:` | 文档更新 | `docs: 更新 API 接口文档` |
| `style:` | 样式调整 | `style: 优化菜谱卡片布局` |
| `refactor:` | 重构 | `refactor: 提取推荐算法为独立函数` |
| `chore:` | 杂项 | `chore: 更新依赖版本` |

## 4. 安全规范

### 4.1 绝对禁止
- ❌ 将 API Key 写入前端代码
- ❌ 将 `.env` 提交到 GitHub
- ❌ 在日志中打印敏感信息
- ❌ 信任前端传来的用户 ID（MVP 阶段暂用 default）

### 4.2 必须执行
- ✅ `.gitignore` 中排除 `.env` 和所有缓存文件
- ✅ 后端校验文件大小（< 10MB）
- ✅ 后端设置每日调用上限
- ✅ 使用 HTTPS 传输（Render 默认提供）
