# 07-面试话术指南

> 本文档帮你准备 AI 产品经理岗位面试中关于「Bonnie Fridge」项目的讲述与答辩。

---

## 🎬 3分钟项目介绍（黄金结构）

### 开场（30秒）：痛点引入

> "面试官您有没有过这种经历——下班打开冰箱，看着一堆食材却不知道做什么菜？
> 我就经常这样。调研后发现，这是都市白领的普遍痛点：食材放到过期、不知道吃什么、做饭时缺调料。
> 所以我做了「Bonnie Fridge · 冰箱侦探」——拍一张冰箱照片，AI 帮你决定今天吃什么。"

### 方案（60秒）：产品闭环

> "产品核心是一个多模态 AI 应用，完整闭环有 4 步：
> 1. **拍**：用户拍冰箱照片，AI 识别出所有食材
> 2. **推**：基于库存 + 口味偏好，推荐‘你现在就能做的菜’
> 3. **查**：也可以主动搜索‘想做红烧肉’，AI 告诉你缺什么
> 4. **买**：自动生成缺料购物清单
> 
> 我还加了两个小功能：语音补充食材（比如‘冷冻层还有牛排’）、做完菜后一键扣减库存。"

### 技术（60秒）：架构与选型

> "技术上我是全栈独立开发的：
> - **前端**：H5 + Tailwind CSS，轻量跨平台
> - **后端**：Python FastAPI，异步高性能
> - **AI**：调用 Kimi K2.7 多模态 API 做图像识别
> - **数据**：25 道家常菜的 JSON 数据库，计算食材匹配度做推荐
> - **部署**：Render 免费服务器
> 
> 这里有个设计我很得意——成本架构。我把 API Key 放在后端环境变量，前端通过代理访问；同时设置每日 10 次限流。这样开源到 GitHub 后，别人 Fork 不会花我的钱，面试官打开就能体验。"

### 收尾（30秒）：成果与反思

> "目前项目已完整跑通，支持拍照识别、推荐、反向查询、购物清单全流程。
> 如果继续做，我会考虑两个方向：
003e 1. 接入向量数据库，把菜谱扩展到 1000+ 道
> 2. 增加用户行为反馈，用协同过滤优化推荐效果
> 
003e 这个项目让我对多模态 AI 的产品化和成本管控有了实际经验。"

## ❓ 10个高频面试问题及回答

### Q1：为什么选这个选题？
**答**："三个原因。第一，这是真实痛点，我自己和身边朋友都有。第二，多模态 AI（图像识别 + 文本生成）是今年的技术热点，面试官能看懂技术深度。第三，可拓展性强——从个人工具可以延伸到社区、电商、智能家居场景。"

### Q2：你的竞品分析怎么做？
**答**："我调研了下厨房、豆果美食等 App。它们的问题是：需要手动输入食材、推荐逻辑基于全网而非你的库存、没有‘反向查询缺料’功能。Bonnie Fridge 的差异化是‘拍一张照就开始’，零输入成本。"

### Q3：推荐算法怎么做的？
**答**："MVP 阶段用的是规则匹配：计算食材覆盖度（匹配度 = 已有食材 / 所需食材）。同时叠加用户画像过滤——比如用户选了‘减脂’，就过滤掉油炸类；选了‘素食’，就只推荐素菜。后续如果数据量大了，会考虑协同过滤 + 内容推荐的混合策略。"

### Q4：API 成本怎么控制？
**答**："三层防护。第一层，后端代理模式，Key 不暴露。第二层，内存限流器，每日最多 10 次免费调用。第三层，图片预处理——上传前用 Pillow 压缩到 512px、转成 JPEG，减少 Token 消耗。如果用户想深度使用，可以自己填 Key 部署，这就是 BYOK 模式。"

### Q5：识别准确率怎么样？
**答**："目前依赖大模型本身的识别能力。我做了两个兜底设计：一是置信度标注（high/medium/low），低置信度项标红提示用户核对；二是支持手动纠偏和语音补充，用户可以随时修正识别结果。产品经理的核心不是追求 100% 自动，而是设计好‘人机协作’的闭环。"

### Q6：如果用户量大了，架构怎么扩展？
**答**："几个方向。数据库从 JSON 文件升级到 PostgreSQL + Redis 缓存。限流器从内存计数升级到 Redis 分布式计数。图片处理加异步队列（Celery）。菜谱数据用向量数据库（Pinecone/Milvus）做语义检索，而不仅是字符串匹配。"

### Q7：用户画像怎么设计的？
**答**："四维度：口味偏好（清淡/微辣/重辣）、饮食目标（减脂/增肌/控糖/素食）、烹饪时长限制（15分钟/30分钟）、常备食材（盐油酱油等默认有）。画像首次设置后自动记忆，每次推荐自动应用。"

### Q8：前后端怎么交互的？
**答**："RESTful API。核心接口有：拍照识别 /api/ingredients/recognize、获取推荐 /api/recipes/recommend、反向查询 /api/recipes/check、生成购物清单 /api/recipes/shopping-list。所有接口统一返回 {success, data, error} 格式。"

### Q9：你做这个项目遇到的最大困难是什么？
**答**："模型兼容性问题。Kimi K2.7-code 是推理模型，它的回答放在 reasoning_content 里而不是 content，导致我最初的 JSON 解析一直失败。解决过程是：加日志打印完整响应体 → 发现 content 为空但 reasoning_content 有数据 → 修改代码同时读取两个字段 → 用正则从推理文字中提取 JSON 数组。这个过程让我深刻理解了‘AI 应用开发就是不断调试 Prompt 和解析响应’。"

### Q10：如果让你商业化，你会怎么做？
**答**："三层变现。第一层，免费版每天 5 次识别。第二层，会员无限次 + 个性化营养报告。第三层，B 端 API 接入智能家居（冰箱内置摄像头，自动推荐菜谱并一键下单买菜）。核心壁垒不是技术，而是用户画像数据和菜谱内容的积累。"

## 🎤 演示建议

### 黄金 2 分钟演示流程
1. **打开线上地址**（提前暖机，避免 Render 休眠等待）
2. **拍一张冰箱照片**上传 → 展示识别结果
3. **点“推荐”** → 展示匹配度、已有/缺料食材
4. **点“我要做这道菜”** → 展示步骤
5. **反向查询“红烧肉”** → 展示缺料分析
6. **点“+清单”** → 展示购物清单可复制

### 注意事项
- **提前准备**：演示前 5 分钟先访问一次线上地址，唤醒 Render 服务
- **备选方案**：如果线上不稳定，本地提前启动好后端，用 `localhost` 演示
- **突出产品思维**：多讲“为什么做这个设计”，少讲技术细节
