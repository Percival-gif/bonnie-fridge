## 2. 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 前端 | H5 + Tailwind CSS + Vanilla JS | 跨平台、零构建、GitHub Pages 直接部署 |
| 后端 | Python + FastAPI | AI 生态友好、异步高性能、自动 API 文档 |
| AI 模型 | Kimi K2.7（Moonshot） | 用户已有 API、国内访问稳定、多模态能力强 |
| 数据库 | 内存 + JSON 文件 | MVP 阶段足够，无需数据库运维 |
| 部署 | Render / Railway | 免费 tier、自动 CI/CD、HTTPS 默认 |

## 3. 安全设计

### 3.1 API Key 保护
- Key 仅存储于后端 `.env` 环境变量
- `.env` 被 `.gitignore` 保护，永不入仓
- 前端通过后端代理间接调用 AI，不接触 Key

### 3.2 限流机制
- 内存计数器限制每日调用次数（默认 10 次）
- 超限返回 429 错误，提示用户次日再来或自部署

### 3.3 跨域配置
- FastAPI CORS 允许所有来源（开发阶段）
- 生产环境建议限制为具体前端域名

## 4. 数据流

```
用户拍照 → 前端压缩图片 → POST /api/ingredients/recognize
    ↓
后端接收 → 校验大小（<10MB）→ 检查限流 → Base64 编码
    ↓
调用 Kimi API（带 Key）→ 解析 JSON 响应 → 存入内存库存
    ↓
返回食材列表 → 前端展示 → 自动触发推荐
    ↓
POST /api/recipes/recommend → 后端计算匹配度 → 返回排序菜谱
    ↓
用户选择菜谱 → 展示步骤 → 确认"做了这道菜"→ 扣减库存
```
